# Database migrations

В проекте настроены миграции MongoDB через `migrate-mongo`.

## Что добавлено

- Конфиг: `migrate-mongo-config.js`
- Папка миграций: `database/migrations`
- Коллекция истории миграций в MongoDB: `db_migrations_changelog`
- Базовая миграция: `20260425190000-create-base-indexes.js`

Базовая миграция:

- создает отсутствующие коллекции `users`, `companies`, `projects`, `tasks`, `chats`, `messages`, `shifts`
- создает индексы, которые уже ожидаются текущими Mongoose-схемами

Индексы:

- `users.email` -> unique index
- `companies.email` -> unique index
- `shifts.workerId`
- `shifts.projectId`
- `shifts.shiftDate`
- `shifts.status`
- `shifts.workerId + shifts.shiftDate + shifts.status`

## Переменные окружения

Обязательная переменная:

- `MONGODB_URI`

Опциональная переменная:

- `MONGODB_DB_NAME` - нужна только если в `MONGODB_URI` не указано имя базы отдельно

Пример:

```env
MONGODB_URI=mongodb://127.0.0.1:27017/project_management
```

или

```env
MONGODB_URI=mongodb+srv://user:password@cluster.mongodb.net/project_management?retryWrites=true&w=majority
```

## Команды

Создать новую миграцию:

```bash
npm run migration:create -- add-project-slug
```

Показать статус:

```bash
npm run migration:status
```

Применить все непримененные миграции:

```bash
npm run migration:up
```

Откатить последнюю примененную миграцию:

```bash
npm run migration:down
```

## Как это работает

`migrate-mongo` хранит список уже выполненных миграций в коллекции `db_migrations_changelog`.

Это значит:

- повторный `migration:up` не запускает уже примененные миграции повторно
- миграции применяются по порядку имен файлов
- `migration:down` откатывает только последнюю примененную миграцию

## Что значит "актуализировать базу" для MongoDB

В этом проекте база данных на MongoDB, поэтому тут нет SQL-таблиц и SQL-колонок в обычном смысле.

Что реально можно и нужно синхронизировать миграцией:

- наличие нужных коллекций
- индексы
- данные existing-документов, если нужен backfill

Что не требуется для новых полей:

- если вы просто добавили поле в Mongoose-схему, отдельная миграция для "создания колонки" не нужна
- новое поле появится в документе тогда, когда документ будет создан или обновлен приложением

Это значит, что для вашего сценария "без создания записей, но чтобы прод соответствовал локальной базе":

- коллекции будут созданы миграцией
- индексы будут созданы миграцией
- сами существующие документы не будут переписываться, если вы отдельно не добавите data migration

## Первый запуск на проде

Перед первым запуском обязательно проверьте данные, потому что базовая миграция создает `unique` индексы на:

- `users.email`
- `companies.email`

Если в базе уже есть дубликаты, миграция завершится ошибкой. Это правильно и безопасно: индекс не должен создаться на плохих данных.

Рекомендуемый порядок:

1. Сделать backup базы.
2. Проверить, что у прод-среды задан корректный `MONGODB_URI`.
3. Установить зависимости:

```bash
npm ci
```

4. Проверить, какие миграции еще не применены:

```bash
npm run migration:status
```

5. Применить миграции:

```bash
npm run migration:up
```

6. После успешного выполнения запускать приложение:

```bash
npm run build
npm run start:prod
```

## Безопасный сценарий деплоя

Если деплой делается вручную на сервере, используйте такой порядок:

```bash
npm ci
npm run build
npm run migration:status
npm run migration:up
npm run start:prod
```

Если приложение запускается через CI/CD, лучше вынести `npm run migration:up` в отдельный deploy step до старта нового инстанса приложения.

## Откат

Если последняя миграция уже применена и нужно откатить только ее:

```bash
npm run migration:down
```

Важно: `down` откатывает только одну последнюю миграцию за запуск. Если нужно откатить несколько, команду запускают несколько раз и каждый раз проверяют `migration:status`.

## Как писать новые миграции

Создайте файл:

```bash
npm run migration:create -- add-company-slug
```

Внутри будет экспорт с двумя функциями:

```js
module.exports = {
  async up(db) {
    // применить изменение
  },

  async down(db) {
    // откатить изменение
  },
};
```

Рекомендации:

- делайте миграции идемпотентными там, где это возможно
- не смешивайте много разных изменений в одной миграции
- если добавляете обязательное поле, сначала заполните старые документы, потом вводите жесткие ограничения
- для опасных миграций всегда делайте backup перед запуском

## Практические замечания для этого проекта

- Сейчас приложение использует MongoDB через Mongoose, поэтому миграции нужны не для SQL-схемы, а для изменения данных, индексов и структуры документов.
- Для продакшена лучше не полагаться только на автоматическое создание индексов через Mongoose, а держать критичные индексы в миграциях.
- Текущая базовая миграция подходит для вашего сценария: она не создает записи, но гарантирует наличие всех основных коллекций и индексов.
- Если захотите, следующим шагом можно вынести создание всех важных индексов и backfill-операции в отдельные именованные миграции под конкретные фичи.
